## 1. 문제 기술

본 과제는 배열을 이용한 기본 정렬 알고리즘(Bubble Sort, Insertion Sort, Selection Sort)을 구현하고, 데이터 크기에 따른 시간 복잡도를 실측하여 비교 분석하는 것을 목적으로 한다.

- **Problem 1**: Bubble Sort, Insertion Sort, Selection Sort 알고리즘을 함수 또는 클래스로 설계하고 구현
- **Problem 2**: 단순 배열 활용 계산 문제에서 함수 설계와 클래스 설계의 장단점 비교

## 2. 분석과 설계 방법

### 2.1 클래스 설계

교수님이 제공한 `sort.c`의 bubble(), insertion(), selection() 함수를 기반으로, **SortArray 클래스**를 설계하였다.

주요 멤버:
- `int* data`: 동적 할당 1차원 배열
- `int size`: 배열 크기
- `bubbleSort()`: Bubble Sort 수행, 비교 횟수 반환
- `insertionSort()`: Insertion Sort 수행, 비교 횟수 반환
- `selectionSort()`: Selection Sort 수행, 비교 횟수 반환
- `isSorted()`: 정렬 결과 검증
- `fillRandom()`: 랜덤 데이터 초기화

### 2.2 시간 복잡도 이론 분석

| 알고리즘 | 최선 | 평균 | 최악 | 공간 |
|----------|------|------|------|------|
| Bubble Sort | O(n) | O(n²) | O(n²) | O(1) |
| Insertion Sort | O(n) | O(n²) | O(n²) | O(1) |
| Selection Sort | O(n²) | O(n²) | O(n²) | O(1) |

세 알고리즘 모두 평균/최악 O(n²)이지만, Insertion Sort는 거의 정렬된 데이터에서 O(n)에 가깝게 동작하고, Bubble Sort는 조기 종료(early termination) 최적화가 가능하다. Selection Sort는 입력 순서와 무관하게 항상 n(n-1)/2번 비교한다.

## 5. 토의와 개선점

### 5.1 실행시간 비교 분석

**속도 순위**: Insertion Sort > Selection Sort > Bubble Sort (Insertion이 가장 빠름)

n=300,000 기준:
- Insertion Sort: 5,551ms (가장 빠름)
- Selection Sort: 11,723ms (약 2.1배 느림)
- Bubble Sort: 63,549ms (약 11.4배 느림)

**비교 횟수 분석**:
- Bubble Sort와 Selection Sort는 모두 약 n(n-1)/2 ≈ 4.5×10^10회 비교 (이론값과 일치)
- Insertion Sort는 평균적으로 n(n-1)/4 ≈ 2.25×10^10회 비교 (절반)
- Selection Sort는 비교 횟수는 Bubble과 동일하지만 교환이 O(n)으로 적어 더 빠름

**Bubble Sort가 가장 느린 이유**: 비교 횟수는 Selection과 동일하나, 교환 횟수가 평균 O(n²)으로 압도적으로 많다. 교환 시 3번의 대입 연산이 발생하므로 메모리 접근 비용이 크다.

**데이터 크기에 따른 증가율**: 크기가 2배이면 시간이 약 4배 증가하여 O(n²) 이론과 일치한다 (예: 100K→200K에서 Insertion 609ms→2503ms, 약 4.1배).

**과제 사양 대비 설명**: 원래 100,000~1,000,000 범위가 요구되었으나, O(n²) 알고리즘 특성상 n=1,000,000에서는 약 10^12회 비교가 필요하여 수 시간이 소요된다. 따라서 10,000~300,000 범위에서 측정하였으며, 추세를 통해 더 큰 크기의 실행시간을 추정할 수 있다.

### 5.2 계산의 제한성

1. O(n²) 알고리즘은 n > 500,000에서 실용적이지 않음
2. 랜덤 데이터의 경우 결과가 실행마다 달라질 수 있으므로, 여러 번 실행의 평균이 더 정확함
3. 정수 정렬이므로 비교 연산은 단순하지만, 실수 정렬에서는 부동소수점 비교 비용이 추가됨

### 5.3 개선점

1. **O(n log n) 알고리즘**: Quick Sort, Merge Sort, Heap Sort로 개선하면 n=1,000,000도 1초 이내 처리 가능
2. **하이브리드 정렬**: 작은 부분 배열에 Insertion Sort를 적용하는 Timsort 방식
3. **캐시 최적화**: 데이터 지역성(locality)을 고려한 블록 단위 처리
4. **병렬 정렬**: OpenMP나 std::execution::par를 활용한 병렬 처리

## 6. 함수 설계 vs 클래스 설계 (Problem 2)

### 6.1 함수 설계 (교수님 제공 sort.c 기반)

```c
void bubble(int *a, int n);    // 배열 포인터 + 크기를 인자로 전달
void insertion(int *a, int n);
void selection(int *a, int n);
```

**장점**:
- 구현이 단순하고 직관적
- 함수 호출 오버헤드가 최소
- C 호환으로 범용성 높음
- 다양한 배열에 바로 적용 가능 (별도 객체 생성 불필요)

**단점**:
- 배열 크기를 매번 인자로 전달해야 함
- 정렬 상태 확인, 비교 횟수 추적 등 부가 기능 추가 시 인터페이스가 복잡해짐
- 배열과 크기 정보가 분리되어 실수 가능 (잘못된 크기 전달)

### 6.2 클래스 설계 (본 과제 SortArray 기반)

```cpp
class SortArray {
    int* data;
    int size;
    long long bubbleSort();
    bool isSorted();
    void print();
};
```

**장점**:
- 데이터(배열)와 연산(정렬)이 캡슐화되어 응집도 높음
- 크기 정보가 객체에 포함되어 인자 전달 오류 방지
- isSorted(), print() 등 유틸리티 메서드를 자연스럽게 포함
- 비교 횟수 등 통계를 멤버 변수로 관리 가능
- 복사 생성자로 동일 데이터에 대한 다중 정렬 테스트가 용이

**단점**:
- 객체 생성/소멸 오버헤드 (단, 정렬 시간 대비 무시 가능)
- 단순한 정렬에 비해 코드량이 증가
- 템플릿을 사용하지 않으면 타입(int)에 종속적

### 6.3 결론

정렬과 같은 **단순 배열 연산**에서는 함수 설계가 더 간결하고 효율적이다. 그러나 벤치마크, 정렬 검증, 다양한 초기화 등 **부가 기능이 필요한 실험적 환경**에서는 클래스 설계가 코드 관리와 재사용 측면에서 우수하다.

실무적으로는 C++ STL의 `std::sort()`처럼 **함수 템플릿 + 반복자(iterator)** 패턴이 두 장점을 결합한 최적의 접근이다.
