## 1. 문제 기술

본 과제는 배열을 이용한 기본 정렬 알고리즘(Bubble Sort, Insertion Sort, Selection Sort)을 구현하고, 데이터 크기에 따른 시간 복잡도를 실측하여 비교 분석하는 것을 목적으로 한다.

- **Problem 1**: Bubble Sort, Insertion Sort, Selection Sort 알고리즘을 함수 또는 클래스로 설계하고 구현
- **Problem 2**: 단순 배열 활용 계산 문제에서 함수 설계와 클래스 설계의 장단점 비교

## 2. 분석과 설계 방법

### 2.1 클래스 설계

교수님이 제공한 `sort.c`의 bubble(), insertion(), selection() 함수를 기반으로, **SortArray 클래스**를 설계하였다.

주요 멤버:
- `int* data`: 동적 할당 1차원 배열
- `int size`: 배열 크기
- `bubbleSort()`: Bubble Sort 수행, 비교 횟수 반환
- `insertionSort()`: Insertion Sort 수행, 비교 횟수 반환
- `selectionSort()`: Selection Sort 수행, 비교 횟수 반환
- `isSorted()`: 정렬 결과 검증
- `fillRandom()`: 랜덤 데이터 초기화

### 2.2 시간 복잡도 이론 분석

| 알고리즘 | 최선 | 평균 | 최악 | 공간 |
|----------|------|------|------|------|
| Bubble Sort | O(n) | O(n²) | O(n²) | O(1) |
| Insertion Sort | O(n) | O(n²) | O(n²) | O(1) |
| Selection Sort | O(n²) | O(n²) | O(n²) | O(1) |

세 알고리즘 모두 평균/최악 O(n²)이지만, Insertion Sort는 거의 정렬된 데이터에서 O(n)에 가깝게 동작하고, Bubble Sort는 조기 종료(early termination) 최적화가 가능하다. Selection Sort는 입력 순서와 무관하게 항상 n(n-1)/2번 비교한다.

## 5. 토의와 개선점

### 5.1 실행시간 및 복잡도 분석

**실험적 데이터와 이론의 일치**:
- 실험 결과 Bubble Sort, Insertion Sort, Selection Sort 모두 데이터 크기가 2배 증가할 때 실행시간이 약 4배 증가하는 $O(n^2)$의 전형적인 곡선을 보였다.
- 특히 Bubble Sort($n=300,000$ 기준 63.5초)가 가장 느렸으며, Insertion Sort(5.5초)가 동일 조건에서 약 11배 이상 빠른 성능을 보였다. 이는 Insertion Sort가 평균적으로 비교 횟수가 적고($n^2/4$) 교환 오버헤드가 작기 때문이다.

**데이터 크기 설정 근거**:
- 과제 안내서의 권장 범위(100,000 ~ 1,000,000) 중 100,000부터 300,000까지 5개 이상의 지점을 집중적으로 측정하였다.
- $n=1,000,000$ 이상의 경우 $O(n^2)$ 알고리즘의 특성상 단일 테스트에 수 시간이 소요되어 실질적인 벤치마크가 어려우나, 300,000까지의 추세선을 통해 $1,000,000$일 때의 시간을 충분히 예측 가능하다는 점에서 타당성을 확보하였다.

**데이터 생성의 객관성**:
- `srand()`와 `rand()` 함수를 조합하여 매 실행 시 서로 다른 무작위 데이터를 생성함으로써 특정 패턴에 유리한 결과가 나오지 않도록 설계하였다.

### 5.2 함수 설계 vs 클래스 설계 (Problem 2)

**함수 설계 (C 스타일)**:
- 장점: 메모리 및 시간 오버헤드가 거의 없으며, 구조가 단순하여 구현이 빠르다.
- 단점: 데이터(배열)와 부가 정보(크기, 통계)가 분리되어 있어 관리 누락의 위험이 크다.

**클래스 설계 (C++ 스타일)**:
- 장점: 데이터와 정렬 알고리즘이 하나의 객체로 캡슐화되어 응집도가 높다. 특히 생성자/소멸자를 통한 **자동 메모리 관리(`new/delete`)**가 가능하여 안정적이다.
- 단점: 아주 미세한 객체 생성 오버헤드가 있으며, 단순 정렬만 필요한 경우 과한 설계일 수 있다.

**결론**: 단순 계산 성능이 최우선인 경우 함수 설계가 유리하나, 본 과제와 같이 다양한 크기를 테스트하고 검증하는 실험 환경에서는 클래스 설계가 코드의 재사용성과 안정성 측면에서 월등히 우수하다.

### 5.3 계산의 제한성 및 개선점

1. **알고리즘적 한계**: $O(n^2)$ 정렬은 대용량 데이터에서 실용성이 낮으므로 Quick Sort($O(n \log n)$) 등으로의 전환이 필요하다.
2. **타입 확장성**: 현재 `int`형에 고정되어 있으나, C++ Template을 활용하면 다양한 데이터 타입에 대응 가능하다.
