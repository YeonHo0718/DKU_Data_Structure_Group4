# 자료구조 과제 #1 보고서
## 모듈라 프로그래밍의 이해와 응용 — 배열과 행렬 연산

---

## 1. 문제 기술

본 과제는 C++ 언어를 활용하여 2차원 행렬의 다양한 연산을 구현하고, 데이터 크기에 따른 시간 복잡도를 분석하는 것을 목적으로 한다. 구체적으로 다음 6개 문제를 다룬다.

- **Problem 1**: 행렬의 덧셈, 뺄셈, 곱셈, element-wise 나눗셈 함수 설계 및 구현. Python numpy와의 비교.
- **Problem 2**: Problem 1의 연산들을 다양한 행렬 크기(10개 이상)에 대해 실행하여 시간 복잡도 분석.
- **Problem 3**: 2차원 행렬의 희소 행렬(Sparse Matrix)과 전치 행렬(Transpose Matrix) 구성.
- **Problem 4**: Problem 3의 연산들에 대한 시간 복잡도 분석.
- **Problem 5**: Problem 2 & 4의 결과 비교 및 설명.
- **Problem 6**: 함수 설계와 클래스 설계의 장단점 비교.

## 2. 분석과 설계 방법

### 2.1 클래스 설계

교수님이 제공한 `gobhagi.cpp`(C++ 클래스 버전)과 `common.c/h`(C 모듈라 버전)를 기반으로, **C++ 클래스 기반** 설계를 채택하였다.

**Matrix 클래스**: 일반 2차원 행렬을 표현하며, 동적 메모리 할당(`new/delete`)으로 임의 크기의 행렬을 지원한다.

주요 멤버:
- `int rows, cols`: 행렬의 크기
- `int** data`: 동적 할당된 2차원 배열
- 정적 메서드: `add()`, `subtract()`, `multiply()`, `divideElementWise()`
- 인스턴스 메서드: `toSparse()`, `transpose()`, `fillRandom()`, `print()`

**SparseMatrix 클래스**: 희소 행렬을 (row, col, value) 삼중쌍(triple)으로 저장한다.

주요 멤버:
- `int rows, cols, numTerms`
- `vector<SparseElement> terms`
- `transpose()`: 열 기준 정렬을 통한 효율적 전치

### 2.2 시간 복잡도 이론 분석

| 연산 | 시간 복잡도 |
|------|------------|
| 덧셈/뺄셈/나눗셈 (M×N) | O(M×N) |
| 곱셈 (M×K) × (K×N) | O(M×K×N) |
| 일반 전치 (M×N) | O(M×N) |
| 희소 변환 (M×N) | O(M×N) |
| 희소 전치 (t개 원소) | O(t + cols) |

## 3. 프로그램 리스트

### 3.1 matrix.h (헤더 파일)

> (include/matrix.h 참조)

- `SparseElement` 구조체: 희소 행렬의 원소 (row, col, value)
- `SparseMatrix` 클래스: 희소 행렬 표현 및 전치
- `Matrix` 클래스: 일반 행렬 + 연산(덧셈, 뺄셈, 곱셈, 나눗셈, 전치, 희소변환)

### 3.2 matrix.cpp (클래스 구현)

> (src/matrix.cpp 참조)

주요 구현 설명:
- **동적 메모리 관리**: 생성자에서 `new`로 할당, 소멸자에서 `delete[]`로 해제. 복사 생성자와 대입 연산자도 구현하여 깊은 복사(deep copy) 보장.
- **곱셈 버그 수정**: 교수님 제공 코드의 `sum += a[r][k] + b[k][c]`를 `sum += A.data[i][k] * B.data[k][j]`로 수정 (덧셈 → 곱셈).
- **0 나눗셈 처리**: element-wise 나눗셈에서 분모가 0인 경우 경고 출력 후 0으로 처리.
- **희소 전치**: 단순 전치가 아닌, 열 카운트 기반 O(n+t) 알고리즘 적용.

### 3.3 main.cpp (메인 드라이버)

> (src/main.cpp 참조)

- `chrono::high_resolution_clock`을 이용한 마이크로초 단위 시간 측정
- 12가지 이상의 다양한 행렬 크기에 대한 자동 벤치마크
- 명령줄 인수로 개별 문제 실행 가능 (`./matrix_ops 1` 등)

### 3.4 Makefile

> (Makefile 참조)

- `g++ -Wall -O2 -std=c++17` 컴파일 옵션
- 분리된 디렉토리 구조 (src → obj → bin)

## 5. 토의와 개선점

### 5.1 Problem 5: 결과 비교 분석

**덧셈/뺄셈/나눗셈**: 모두 O(M×N)의 시간 복잡도를 보인다. 행렬 크기가 2배가 되면 실행시간이 약 4배 증가하는 것을 확인할 수 있다 (예: 500×500 → 1000×1000에서 Add가 0.466ms → 1.630ms, 약 3.5배). 덧셈이 뺄셈보다 다소 느린 이유는 결과 행렬의 메모리 할당 시간이 포함되기 때문이다.

**곱셈**: O(M×K×N)의 3차 시간 복잡도를 보인다. 크기가 2배가 되면 약 8배 증가하여 이론과 일치한다 (500→1000에서 43ms → 417ms, 약 9.7배).

**일반 전치 vs 희소 전치**: 일반 전치는 O(M×N)이지만, 희소 전치는 O(t+cols)로 비영 원소 수에 비례한다. 밀도 10%에서 1000×1000 행렬의 경우, 일반 전치(0.946ms)에 비해 희소 전치(0.236ms)가 약 4배 빠르다. 밀도가 낮을수록 희소 행렬의 이점이 커진다.

**계산의 제한성**: 정수 나눗셈에서 소수점 이하가 절삭되므로 정밀도 손실이 발생한다. 실수 연산이 필요한 경우 `double` 타입으로 확장해야 한다.

### 5.2 테스트 결과 검증

- 3×3 소규모 행렬에서 수동 계산과 프로그램 결과가 일치함을 확인
- 희소 행렬과 일반 전치의 결과가 동일함을 확인
- 교수님 제공 코드의 곱셈 연산 오류(`+` → `*`) 수정 반영

### 5.3 개선점

1. **Strassen 알고리즘**: 현재 O(n³) 곱셈을 O(n^2.807)로 개선 가능
2. **캐시 최적화**: 전치 후 곱셈(B를 전치하여 행 방향 접근)으로 캐시 적중률 향상
3. **실수 타입 지원**: template 클래스로 int/float/double 통합
4. **병렬 처리**: OpenMP를 활용한 행렬 연산 병렬화
5. **메모리 연속 할당**: 현재 행 단위 개별 할당을 하나의 연속 블록으로 변경하면 캐시 성능 향상

## 6. 함수 설계 vs 클래스 설계 (Problem 6)

### 6.1 함수 설계 (C 스타일 — 교수님 제공 `common.c/h` 기반)

**장점**:
- 구조가 단순하고 오버헤드가 적음
- C 언어 호환으로 임베디드 등 제약 환경에서 활용 가능
- 함수 단위 테스트가 직관적

**단점**:
- 행렬 크기(rows, cols)를 매번 인자로 전달해야 함
- 메모리 할당/해제를 사용자가 직접 관리해야 하므로 메모리 누수 위험
- 관련 연산들이 분산되어 코드 응집도가 낮음

### 6.2 클래스 설계 (C++ 스타일 — 본 과제 구현)

**장점**:
- 데이터(data, rows, cols)와 연산(add, multiply 등)이 하나의 클래스에 캡슐화
- 생성자/소멸자로 메모리 자동 관리 → 메모리 누수 방지
- 연산자 오버로딩(`+`, `-`, `*`)으로 직관적 코드 작성 가능
- 상속을 통해 SparseMatrix 등 파생 클래스 설계 용이

**단점**:
- 클래스 설계의 학습 곡선이 있음
- 복사 생성자, 대입 연산자 등 추가 구현 필요
- 가상 함수 사용 시 vtable로 인한 약간의 성능 오버헤드

### 6.3 결론

행렬 연산의 경우, 데이터와 연산이 밀접하게 결합되어 있으므로 **클래스 설계가 더 적합**하다. 특히 메모리 관리의 안전성과 코드의 재사용성 측면에서 클래스 설계의 이점이 크다. 단, 성능이 최우선인 경우(대규모 과학 계산 등)에는 함수 기반 설계에 최적화된 라이브러리(BLAS, LAPACK 등)를 활용하는 것이 바람직하다.
