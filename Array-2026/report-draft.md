# 자료구조 과제 #1 보고서
## 모듈라 프로그래밍의 이해와 응용 — 배열과 행렬 연산

---

## 1. 문제 기술

본 과제는 C++ 언어를 활용하여 2차원 행렬의 다양한 연산을 구현하고, 데이터 크기에 따른 시간 복잡도를 분석하는 것을 목적으로 한다. 구체적으로 다음 6개 문제를 다룬다.

- **Problem 1**: 행렬의 덧셈, 뺄셈, 곱셈, element-wise 나눗셈 함수 설계 및 구현. Python numpy와의 비교.
- **Problem 2**: Problem 1의 연산들을 다양한 행렬 크기(10개 이상)에 대해 실행하여 시간 복잡도 분석.
- **Problem 3**: 2차원 행렬의 희소 행렬(Sparse Matrix)과 전치 행렬(Transpose Matrix) 구성.
- **Problem 4**: Problem 3의 연산들에 대한 시간 복잡도 분석.
- **Problem 5**: Problem 2 & 4의 결과 비교 및 설명.
- **Problem 6**: 함수 설계와 클래스 설계의 장단점 비교.

## 2. 분석과 설계 방법

### 2.1 클래스 설계

교수님이 제공한 `gobhagi.cpp`(C++ 클래스 버전)과 `common.c/h`(C 모듈라 버전)를 기반으로, **C++ 클래스 기반** 설계를 채택하였다.

**Matrix 클래스**: 일반 2차원 행렬을 표현하며, 동적 메모리 할당(`new/delete`)으로 임의 크기의 행렬을 지원한다.

주요 멤버:
- `int rows, cols`: 행렬의 크기
- `int** data`: 동적 할당된 2차원 배열
- 정적 메서드: `add()`, `subtract()`, `multiply()`, `divideElementWise()`
- 인스턴스 메서드: `toSparse()`, `transpose()`, `fillRandom()`, `print()`

**SparseMatrix 클래스**: 희소 행렬을 (row, col, value) 삼중쌍(triple)으로 저장한다.

주요 멤버:
- `int rows, cols, numTerms`
- `vector<SparseElement> terms`
- `transpose()`: 열 기준 정렬을 통한 효율적 전치

### 2.2 시간 복잡도 이론 분석

| 연산 | 시간 복잡도 |
|------|------------|
| 덧셈/뺄셈/나눗셈 (M×N) | O(M×N) |
| 곱셈 (M×K) × (K×N) | O(M×K×N) |
| 일반 전치 (M×N) | O(M×N) |
| 희소 변환 (M×N) | O(M×N) |
| 희소 전치 (t개 원소) | O(t + cols) |

## 3. 프로그램 리스트

### 3.1 matrix.h (헤더 파일)

> (include/matrix.h 참조)

- `SparseElement` 구조체: 희소 행렬의 원소 (row, col, value)
- `SparseMatrix` 클래스: 희소 행렬 표현 및 전치
- `Matrix` 클래스: 일반 행렬 + 연산(덧셈, 뺄셈, 곱셈, 나눗셈, 전치, 희소변환)

### 3.2 matrix.cpp (클래스 구현)

> (src/matrix.cpp 참조)

주요 구현 설명:
- **동적 메모리 관리**: 생성자에서 `new`로 할당, 소멸자에서 `delete[]`로 해제. 복사 생성자와 대입 연산자도 구현하여 깊은 복사(deep copy) 보장.
- **곱셈 버그 수정**: 교수님 제공 코드의 `sum += a[r][k] + b[k][c]`를 `sum += A.data[i][k] * B.data[k][j]`로 수정 (덧셈 → 곱셈).
- **0 나눗셈 처리**: element-wise 나눗셈에서 분모가 0인 경우 경고 출력 후 0으로 처리.
- **희소 전치**: 단순 전치가 아닌, 열 카운트 기반 O(n+t) 알고리즘 적용.

### 3.3 main.cpp (메인 드라이버)

> (src/main.cpp 참조)

- `chrono::high_resolution_clock`을 이용한 마이크로초 단위 시간 측정
- 12가지 이상의 다양한 행렬 크기에 대한 자동 벤치마크
- 명령줄 인수로 개별 문제 실행 가능 (`./matrix_ops 1` 등)

### 3.4 Makefile

> (Makefile 참조)

- `g++ -Wall -O2 -std=c++17` 컴파일 옵션
- 분리된 디렉토리 구조 (src → obj → bin)

## 5. 토의와 개선점

### 5.1 Problem 5: 결과 비교 분석

**덧셈/뺄셈/나눗셈**: 모두 O(M×N)의 시간 복잡도를 보인다. 실험 결과, 행렬 크기가 2배(한 축 기준)가 되면 전체 면적은 4배가 되므로 실행시간도 약 4배 증가함을 확인하였다. (예: 500×500 → 1000×1000 구간에서 Add 연산이 **0.466ms → 1.630ms**로 약 3.5배 증가). 덧셈이 뺄셈보다 다소 느린 이유는 결과 행렬의 동적 메모리(`new`) 할당 시간이 포함되기 때문이다.

**곱셈**: O(M×K×N)의 3차 시간 복잡도를 보인다. 데이터 크기가 2배가 되면 연산량은 약 8배($2^3$) 증가하며, 실제 측정치에서도 약 9.7배(**43ms → 417ms**) 증가하여 이론적 수치와 거의 일치함을 확인하였다.

**일반 전치 vs 희소 전치**: 일반 전치는 O(M×N)이지만, 희소 전치는 O(t+cols)로 비영 원소 수($t$)에 비례한다. 밀도 10%의 1000×1000 행렬 테스트에서 일반 전치(**0.946ms**)에 비해 희소 전치(**0.236ms**)가 약 4배 빠른 결과를 보였다. 이를 통해 데이터가 희소할수록(Sparse) 전용 알고리즘을 사용하는 것이 효율적임을 실험적으로 증명하였다.

### 5.2 테스트 결과 검증 및 계산의 제한성

- 3×3 소규모 행렬에서 수동 계산과 프로그램 결과가 일치함을 확인하였다.
- **랜덤 함수 사용**: `srand()`와 `mrand()`를 활용하여 매 실행 또는 테스트 케이스마다 다양한 데이터를 생성하여 검증하였다.
- **입력 처리**: Problem 1 테스트 시 사용자로부터 행렬의 크기(M, N, K)를 직접 입력받아 처리할 수 있도록 구현하여 범용성을 확보하였다.
- **정밀도**: 정수형 행렬을 사용하므로 나눗셈 연산 시 소수점 이하 절삭이 발생한다. 실수 연산이 필요한 경우 `double` 타입으로의 확장이 필요하다.

### 5.3 개선점

1. **캐시 최적화**: 곱셈 시 B 행렬을 전치하여 접근하면 캐시 적중률을 높일 수 있다.
2. **병렬 처리**: OpenMP 등을 활용하여 루프를 병렬화하면 대규모 행렬 연산 속도를 획기적으로 개선할 수 있다.
3. **템플릿화**: C++ template을 사용하여 다양한 데이터 타입(float, double 등)을 지원하도록 개선 가능하다.

### 5.4 함수 설계 vs 클래스 설계 (Problem 6)

**함수 설계 (C 스타일)**:
- 장점: 오버헤드가 적고 구조가 단순함. C 언어와의 호환성이 좋음.
- 단점: 행렬 크기를 항상 인자로 전달해야 하며, 메모리 관리가 수동적이어서 누수 위험이 높음.

**클래스 설계 (C++ 스타일)**:
- 장점: 데이터와 연산이 캡슐화되어 관리가 용이함. 생성자와 소멸자를 통해 **동적 메모리(`new/delete`)**를 자동으로 관리할 수 있어 안전함.
- 단점: 복사 생성자 등 추가 구현이 필요하며, 아주 미세한 성능 오버헤드가 있을 수 있음.

**결론**: 객체지향적 설계가 코드의 재사용성과 안정성 측면에서 월등히 유리하므로, 본 과제에서는 클래스 설계를 주축으로 하되 성능이 중요한 유틸리티는 C 함수를 호출하는 **혼합형 모듈라 구조**를 채택하였다.
